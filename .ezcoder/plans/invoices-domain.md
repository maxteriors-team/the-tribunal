# Invoices domain — implementation plan

## Goal
Add a workspace-scoped **customer invoicing** vertical to The Tribunal: create/send
invoices with line items, take payment via Stripe, and reconcile paid/overdue
status. This is one of the six target CRM domains (lead tracking, dispatching,
invoices, messaging, voice agents, client database) and is the only one with **no
backend domain** today.

## Current state (verified)
- **Greenfield backend.** No `Invoice` model/schema/service/router exists.
- **Dead UI stub already present:** `frontend/src/components/actions/quick-actions-section.tsx`
  has a `send_invoice` quick action (`agent.ts` action type `"send_invoice"`) that
  only fires a toast — no backend call. The plan wires this up.
- `frontend/src/lib/query-keys.ts:191` `billing.invoices` is **SaaS subscription**
  billing (Stripe portal), *not* customer invoices. Do not overload it; add a new
  `invoices` key namespace.
- **Strong local templates to mirror (do not invent patterns):**
  - `backend/app/models/opportunity.py` — `Opportunity` + `OpportunityLineItem`
    (quantity / unit_price / discount / total via `Numeric`) + `OpportunityActivity`.
    Invoice + InvoiceLineItem map almost 1:1 onto this.
  - `backend/app/services/payments/call_payment_service.py` — Stripe Checkout
    Session helpers: `is_payment_configured()`, `to_minor_units(amount, currency)`,
    `create_payment_checkout_session(...)`, `retrieve_session_status(...)`. Reuse
    these; do not write new Stripe plumbing.
  - `backend/app/api/v1/opportunities.py` — router shape, deps, pagination,
    line-item sub-resource endpoints. Mirror endpoint layout.
  - `backend/app/api/webhooks/` (Stripe/Telnyx/Resend) — webhook handler +
    signature-verification + idempotency conventions for the payment webhook.
- Alembic head: `a0e8a88f7801`. Use `make migrate.new m="add invoices"` to
  autogenerate; never hand-pick revision ids.
- Money convention: `Numeric(12, 2)` for totals, `Numeric(10, 2)` for unit/qty,
  `currency` as `String(3)`. Stripe amounts derived at request time via
  `to_minor_units`.

## Data model (`backend/app/models/invoice.py`)
Mirror the Opportunity trio. Workspace-scoped, contact-linked.

**`Invoice` (`invoices`)**
- `id` UUID pk; `workspace_id` FK→workspaces CASCADE, indexed; `contact_id`
  FK→contacts (bill-to), indexed. **Gotcha:** `contacts.id` is `BigInteger`
  (int, autoincrement) — not UUID — so `contact_id` is `Mapped[int]`
  (mirror `Opportunity.primary_contact_id`).
- `number` — human invoice number, unique **per workspace** (e.g. `INV-000123`).
  Generate via a per-workspace counter; enforce `UniqueConstraint(workspace_id, number)`.
- Optional links to existing domains: `opportunity_id` FK→opportunities SET NULL
  (invoice a won deal). (Skip `job_id` — the jobs/field-service domain lives on a
  different branch and isn't on `main`.)
- Money: `subtotal`, `tax_amount`, `discount_amount`, `total` `Numeric(12,2)`;
  `amount_paid` `Numeric(12,2)` default 0; `currency` `String(3)` default `"USD"`.
- `status` Enum `invoice_status`: `draft, sent, paid, partial, void, overdue`
  (default `draft`, indexed). Status is **derived** from amount_paid + due_date by
  the service (mirror the opportunity status approach), not free-set by clients.
- Dates: `issue_date` DATE, `due_date` DATE, `sent_at`/`paid_at` timestamptz nullable.
- `notes` / `terms` Text nullable.
- Stripe: `stripe_checkout_session_id`, `stripe_payment_intent_id` String nullable,
  indexed (for webhook reconciliation lookups).
- `created_by_id` FK→users SET NULL; `created_at`/`updated_at` timestamptz.
- relationships: `line_items` (cascade all, delete-orphan), `contact`, `workspace`.

**`InvoiceLineItem` (`invoice_line_items`)** — copy `OpportunityLineItem`:
`id`, `invoice_id` FK CASCADE indexed, `name`, `description`, `quantity`
`Numeric(10,2)` default 1, `unit_price` `Numeric(10,2)`, `discount`
`Numeric(10,2)` default 0, `total` `Numeric(12,2)`, timestamps. `total` computed
server-side (`quantity*unit_price - discount`); never trust client total.

**`InvoiceActivity` (`invoice_activities`)** *(optional, phase 2)* — audit trail
mirroring `OpportunityActivity` (`created, sent, viewed, paid, voided`).

Register all in `backend/app/models/__init__.py` (import + `__all__`) exactly like
the Opportunity entries.

## Schemas (`backend/app/schemas/invoice.py`)
Mirror `schemas/opportunity.py`: `InvoiceLineItemCreate/Update/Response`,
`InvoiceCreate` (with nested line items), `InvoiceUpdate`, `InvoiceResponse`,
`InvoiceDetailResponse` (includes line items + contact summary),
`PaginatedInvoices`. Server computes totals; `status`, `number`, `amount_paid`,
`total` are **response-only** (not client-writable).

## Service (`backend/app/services/invoices/invoice_service.py`)
Mirror `services/opportunities/opportunity_service.py`. Methods:
- `create` (allocates `number` per workspace, computes line totals + subtotal/total),
  `get`/`list` (workspace-scoped, paginated, filter by status/contact/date),
  `update`, `delete` (block delete once `sent`/`paid` → void instead),
  `add_line_item`/`update_line_item`/`remove_line_item` (recompute totals),
  `mark_sent` (sets `sent_at`, status→`sent`, triggers email),
  `derive_status(invoice)` (paid/partial/overdue from amount_paid + due_date),
  `record_payment(invoice, amount, payment_intent_id)` (idempotent; bumps
  `amount_paid`, re-derives status, sets `paid_at`).
- `create_payment_link(invoice)` → calls the keyword-only
  `call_payment_service.create_payment_checkout_session(amount=<balance due,
  major units>, currency=..., product_name=f"Invoice {number}",
  metadata={"invoice_id": ..., "workspace_id": ...},
  customer_email=<bill-to contact email>)`; stores the returned
  `stripe_checkout_session_id`. Guard with `is_payment_configured()` (clean 4xx
  when Stripe unset). The helper takes **major-unit** amounts and converts via
  `to_minor_units` internally — pass dollars, not cents.
- Reuse `app.services.contacts.contact_filters` only if filtering by contact rules.

## API (`backend/app/api/v1/invoices.py`)
Mirror `opportunities.py`; register in `app/api/v1/router.py` with
`prefix="/workspaces/{workspace_id}/invoices"`. Endpoints:
- `GET ""` (paginated list, filters), `POST ""` (create, 201),
  `GET /{id}` (detail), `PUT /{id}`, `DELETE /{id}` (204; void rules),
- line items: `POST /{id}/line-items`, `PUT /{id}/line-items/{item_id}`,
  `DELETE /{id}/line-items/{item_id}`,
- `POST /{id}/send` (email the invoice), `POST /{id}/payment-link` (Stripe Checkout
  URL), `GET /{id}/pdf` *(phase 2 — reuse `pypdf`/existing PDF tooling)*.
- All workspace-scoped via existing deps (`get_workspace`, `CurrentUser`); enforce
  member-role gating consistent with opportunities.

## Stripe payment webhook (`backend/app/api/webhooks/stripe_*`)
- Add `checkout.session.completed` / `payment_intent.succeeded` handling that looks
  up the invoice by `stripe_payment_intent_id` / session id and calls
  `record_payment(...)`. **Idempotent** (Stripe retries) — mirror existing webhook
  idempotency (`app.services.idempotency`). Verify signature like other webhooks.
- Confirm whether a Stripe webhook router already exists for call payments and
  extend it rather than adding a second endpoint.

## Open decisions (resolve at build time, no blocking)
- Invoice number format/sequence (per-workspace counter vs ULID-ish). Default:
  zero-padded per-workspace counter `INV-000001`.
- Tax handling: single `tax_amount` now vs per-line tax later. Default: invoice-level
  `tax_amount` in phase 1.
- Partial payments: supported via `amount_paid` + `partial` status from day one.
- PDF + activity log are phase 2 (don't block payments).
